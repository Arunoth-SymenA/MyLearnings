# When to use Release Management in DevOpsArk

Use Release Management when your team needs to plan, coordinate, and execute releases in a controlled and structured way.

## Best-fit scenarios

- You need to organize a new release with defined timelines and ownership.
- You want to assign and track tasks across teams before deployment.
- You need approval workflows before executing release activities.
- You want visibility into release progress and task status.
- You need to manage both planned (Normal) and urgent (Emergency) releases.

## When it is most useful in delivery lifecycle

- **Release planning:** define scope, timelines, and responsibilities.
- **Pre-release coordination:** assign tasks and ensure readiness through acceptance.
- **Execution phase:** track progress while the release is in progress.
- **Post-release closure:** finalize release status as complete or incomplete.
- **Operational governance:** enforce structured workflows across teams.

## When to use each primary action

- **Create Release**  
  When a new release cycle needs to be initiated with defined schedule and scope.

- **Add Tasks**  
  When breaking down release work into actionable items for teams or individuals.

- **Acceptance Mail**  
  When task owners need to acknowledge and accept assigned responsibilities.

- **Confirmation Mail**  
  When all tasks are accepted and the release is ready for execution.

- **Start Release (In Progress)**  
  When execution begins after all confirmations are completed.

- **Close Release**  
  When the release work is finished and needs to be marked as complete or incomplete.

## Practical rule of thumb

If the task is “plan, track, and execute a release with clear ownership, approvals, and status visibility,” Release Management is the right place to do it.
